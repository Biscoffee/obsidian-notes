面向有工程背景的开发者，系统规划从 LLM 到 Agent Harness 的学习路径。明确哪些概念只需了解、哪些需要深入理解、哪些必须动手实践，并为每个实践环节设计具体的 Demo 项目。

核心原则：不要自下而上从理论学起，而是从中间切入动手做，遇到瓶颈再回头补底层知识。

一、三层学习模型

|   |   |   |   |
|---|---|---|---|
|层次|目标|投入比例|包含概念|
|了解层|知道原理和边界，能在技术讨论中不露怯|~10%|Transformer 原理、Fine-tuning/RLHF/DPO、Scaling Laws|
|理解层|能做技术选型和方案评审，知道什么场景用什么|~30%|LLM 能力边界、Prompt Engineering、RAG、Multi-Agent|
|实践层|动手写代码、反复调试、积累工程经验|~60%|Tool Use、AI Agent、MCP、Context Engineering、Harness Engineering|

二、了解层：知道是什么就够（~10% 时间）

2.1 Transformer 架构原理

学到什么程度：能用大白话解释 Self-Attention 是什么，知道为什么 Decoder-only 是主流，不需要手推公式。


为什么不需要深入：除非你要从零训练模型或做底层推理优化，否则 Transformer 的数学细节不影响你的日常开发。就像你用 MySQL 不需要懂 B+ 树的每个实现细节。

学习清单：

|   |   |   |
|---|---|---|
|内容|推荐资源|时间|
|Self-Attention 直觉理解|3Blue1Brown "Attention in Transformers" (YouTube)|30 min|
|LLM 如何工作的全景|Andrej Karpathy "Intro to LLMs" (YouTube)|1 hr|
|Token 是什么|在 OpenAI Tokenizer 网站上亲手试几个例子|15 min|

你只需要记住的核心知识点：

• LLM = 一个超大的"文字接龙"程序，每次预测下一个 Token

• Token ≠ 字 ≠ 词，中文大约 1 个字 = 1.5-2 个 Token

• 上下文窗口 = 模型一次能看到的最大 Token 数，不是无限的记忆

• Decoder-only 架构（GPT/Claude/Llama）是当前主流，因为生成效率最高

2.2 Fine-tuning / RLHF / DPO

学到什么程度：知道什么时候该用微调、LoRA 是什么原理、RLHF 是怎么让模型"听话"的。

为什么不需要深入：对于 90% 的 AI 应用，好的 Prompt + RAG + Agent 完全够用，不需要亲自微调模型。微调是最后手段，不是第一选择。

学习清单：

|   |   |   |
|---|---|---|
|内容|推荐资源|时间|
|微调 vs RAG 怎么选|Anthropic 官方文档 "When to Fine-tune"|20 min|
|LoRA 原理图解|搜索 "LoRA explained simply" 博客|30 min|
|RLHF 流程理解|HuggingFace Blog "RLHF explained"|30 min|

你只需要记住的决策框架：

需求是什么？

├── 需要最新知识/企业私有数据 → 用 RAG，不用微调

├── 需要特定回答风格/格式 → 先试 Prompt，不够再微调

├── 需要教模型一个全新技能 → 考虑 LoRA 微调

└── 需要模型行为更安全/对齐 → 这是模型厂商做的（RLHF/DPO）

2.3 Scaling Laws

学到什么程度：知道"更大≠更好，要看性价比"就够了。

一句话总结：模型性能与参数量、数据量、计算量成幂律关系，但存在边际递减。实际选模型时，要根据任务复杂度和成本做平衡（比如简单任务用 Haiku/Mini 级别就够了，不需要上 Opus）。

三、理解层：能做技术选型（~30% 时间）

3.1 LLM 能力边界

学到什么程度：能准确判断"这个任务 LLM 能不能做""应该用什么级别的模型""需要哪些辅助技术"。

为什么重要：这是所有后续技术决策的基础。不理解 LLM 的边界，就会出现"能用 RAG 解决的问题非要微调"或"Agent 做不了的事硬让 Agent 做"。

学习方式：

|   |   |   |
|---|---|---|
|内容|方式|时间|
|各模型能力对比|阅读 Chatbot Arena 排行榜、LMSYS 评测|1 hr|
|幻觉问题理解|亲手测试：问模型一些你知道答案的专业问题，看它错在哪|1 hr|
|模型选型实践|同一个任务分别用 Claude Haiku / Sonnet / Opus 做，对比质量和成本|2 hr|
|上下文窗口测试|给模型一个很长的文档，测试它对不同位置信息的记忆力|1 hr|

你需要建立的判断力：

|   |   |   |
|---|---|---|
|任务类型|LLM 表现|是否需要辅助|
|创意写作、头脑风暴|很强|不需要|
|代码生成（常见语言）|很强|不需要|
|实时信息查询|做不到|需要 Tool Use / RAG|
|精确数学计算|不可靠|需要 Tool Use（代码执行）|
|企业私有知识问答|不知道|需要 RAG|
|多步复杂任务|单次做不好|需要 Agent|
|长时间任务|会"走神"|需要 Context Engineering|

3.2 Prompt Engineering

学到什么程度：能写出高质量的 System Prompt，理解结构化输入对模型的影响，掌握 CoT 等核心技巧。

为什么不归入实践层：在 Agent 时代，单独的 Prompt 优化重要性在下降——它被"吸收"进了 System Prompt 设计和工具描述设计中。你不需要成为 Prompt 大师，但底层功力必须扎实。

学习方式：

|   |   |   |
|---|---|---|
|内容|方式|时间|
|核心技巧（CoT/Few-shot/Role）|阅读 OpenAI Prompt Engineering Guide|2 hr|
|System Prompt 设计|研究 Claude Code / Cursor 的 System Prompt 泄露版本|2 hr|
|结构化输入实践|对比用纯文本 vs XML 标签 vs Markdown 给模型输入，观察效果差异|1 hr|

练习项目：

任务：为一个"代码审查 Agent"设计 System Prompt

要求：

1. 定义角色（资深工程师，10年经验）

2. 定义输入格式（Git diff 格式）

3. 定义输出格式（问题列表 + 严重级别 + 修改建议）

4. 定义行为约束（不确定时说明原因，不擅自修改代码）

5. 定义错误处理（输入不是有效 diff 时怎么办）

验证：用这个 Prompt 让模型审查 3 个真实的 PR diff，评估输出质量


3.3 RAG 检索增强生成

学到什么程度：理解完整架构，能做技术选型（嵌入模型、向量数据库、分块策略），知道 RAG 的典型失败模式和优化方向。

为什么不归入实践层：RAG 已经是非常成熟的技术，有大量现成的框架和服务。除非你的核心工作就是搭建 RAG 系统，否则理解原理 + 会调参就够了。Agent 的 RAG 能力通常由框架/平台提供。

学习方式：






|          |                               |      |
| -------- | ----------------------------- | ---- |
| 内容       | 方式                            | 时间   |
| RAG 完整流程 | 跟着 LangChain RAG Tutorial 走一遍 | 3 hr |
| 分块策略对比   | 对同一份文档用不同分块策略，对比检索效果          | 2 hr |
| 典型失败模式   | 故意测试边界：跨段落的问题、需要推理的问题、模糊查询    | 1 hr |

练习项目：

任务：搭建一个"公司技术文档问答系统"

步骤：

1. 准备 10-20 篇技术文档（Markdown/PDF）

2. 选择分块策略（推荐 RecursiveCharacterTextSplitter，1000 字符 + 200 重叠）

3. 选择嵌入模型（推荐 BGE-M3 或 OpenAI text-embedding-3-large）

4. 选择向量数据库（推荐 Chroma 用于学习）

5. 构建检索链，测试 10 个问题

6. 对比加 Reranker 前后的效果

验证指标：

- 检索准确率：Top-5 中包含正确答案的比例

- 回答质量：是否准确引用了原文

- 失败案例分析：哪些问题答错了，为什么

3.4 Multi-Agent

学到什么程度：理解 5 种协作模式，重点掌握 Sub-Agent 模式，能判断是否真的需要 Multi-Agent。

为什么归入理解层而非实践层：过度设计 Multi-Agent 是当前最常见的反模式。很多团队搞了一堆 Agent 互相聊天，结果还不如一个设计良好的单 Agent。先把单 Agent 做好，证明它不够了再上 Multi-Agent。

核心判断框架：

我需要 Multi-Agent 吗？

├── 任务能拆成完全独立的子任务吗？

│   ├── 能 → 考虑并行 Multi-Agent

│   └── 不能 → 先用单 Agent

├── 单 Agent 上下文会溢出吗？

│   ├── 会 → 用 Sub-Agent 模式隔离上下文

│   └── 不会 → 不需要 Multi-Agent

└── 需要不同"专业角色"吗？

    ├── 需要（如写代码+审代码） → 考虑串行 Multi-Agent

    └── 不需要 → 不需要 Multi-Agent

你需要重点掌握的：Sub-Agent 模式（这是最实用、最不容易出错的 Multi-Agent 形式）

四、实践层：必须动手写代码（~60% 时间）

4.1 Tool Use — 实践项目

为什么是重点：你的 Agent 有多强，80% 取决于你的工具设计得有多好。 工具的 description 写得好不好、参数设计是否合理、错误信息是否对 LLM 友好——这些决定了 Agent 的成败。

实践 Demo 1：个人效率工具集

目标：为 LLM 定义一套个人效率工具，让它能帮你完成日常任务

需要实现的工具：

┌─────────────────────────────────────────────────┐

│ Tool 1: web_search(query) → 搜索互联网            │

│ Tool 2: read_url(url) → 读取网页内容               │

│ Tool 3: run_python(code) → 执行 Python 代码        │

│ Tool 4: read_file(path) → 读取本地文件              │

│ Tool 5: write_file(path, content) → 写入文件       │

└─────────────────────────────────────────────────┘

练习重点：

1. 为每个工具写 description（反复打磨！）

2. 测试：给 LLM 一个任务"搜索 xxx 的信息，整理后保存到文件"

3. 观察 LLM 是否正确选择工具、传参是否正确

4. 故意测试错误情况：URL 无效、文件不存在、Python 报错

5. 优化工具的错误返回信息，让 LLM 能理解并重试

关键学习点：

- description 差一个词，模型行为可能完全不同

- 工具数量宜少不宜多（5-10个是甜蜜点）

- 好的错误信息 = LLM 能自动恢复

实践 Demo 2：做减法实验

目标：验证"少即是多"的工具设计原则

实验：

1. 先给 Agent 提供 20 个工具，测试 10 个任务，记录成功率

2. 删掉功能重叠的工具，保留 10 个，重新测试

3. 继续精简到 5 个核心工具，重新测试

4. 对比三组的成功率

预期结论：工具越少（但描述越精准），Agent 表现越好

4.2 AI Agent — 实践项目

为什么是重点：Agent 是当前 AI 应用的主力形态。不管你将来做什么 AI 产品，核心都是 Agent 循环。

实践 Demo 3：从零构建一个调研 Agent

目标：构建一个能自主完成技术调研的 Agent

功能需求：

  输入："调研 2025 年主流的 AI Agent 框架"

  输出：一份结构化的调研报告（Markdown 格式）

Agent 需要自主完成的步骤：

  1. 分析调研主题，制定搜索计划

  2. 搜索相关信息（多轮搜索）

  3. 阅读关键网页，提取信息

  4. 整理对比信息

  5. 生成结构化报告

  6. 保存到文件

技术选型：

  - 框架：Claude Agent SDK（推荐入门）或 LangGraph（推荐进阶）

  - 工具：web_search + read_url + write_file

  - 模型：Claude 3.5 Sonnet（性价比最高）

代码骨架（Claude Agent SDK）：

import anthropic

client = anthropic.Anthropic()

tools = [search_tool, browse_tool, file_tool]  # 你在 Demo 1 中已实现

messages = [{"role": "user", "content": "调研 2025 年主流 AI Agent 框架"}]

# Agent 循环

for i in range(20):  # 最多 20 轮

    response = client.messages.create(

        model="claude-sonnet-4-20250514",

        max_tokens=4096,

        system="你是一位技术调研助手...",

        tools=tools,

        messages=messages

    )

    # 检查是否需要调用工具

    if response.stop_reason == "tool_use":

        # 执行工具，将结果加入 messages

        for block in response.content:

            if block.type == "tool_use":

                result = execute_tool(block.name, block.input)

                messages.append(...)  # 添加工具结果

    elif response.stop_reason == "end_turn":

        print("Agent 完成任务")

        break

验证标准：

- Agent 能在 10 轮内完成调研

- 报告包含至少 3 个框架的对比

- 信息准确（与你手动搜索对比）

- 报告结构清晰（有标题、表格、优劣分析）

进阶挑战：

- 加入 max_budget 控制（限制 token 使用量）

- 加入超时控制（超过 5 分钟自动停止）

- 加入日志记录（记录每一步的思考和行动）

4.4 Context Engineering — 实践项目

为什么是最高优先级之一：Agent 失败的 80% 原因不是模型不够聪明，而是上下文管理不当。这是从 Demo 到 Production 的关键门槛。

实践 Demo 7：上下文溢出实验

目标：亲身体验上下文管理的重要性

实验步骤：

1. 创建一个 Agent，给它一个"系统规则"：

   "你的名字叫小助手，回答问题时必须先说'你好，我是小助手'"

2. 不做任何上下文管理，连续与它对话 50 轮

3. 记录从第几轮开始，它"忘记"了自己的名字和规则

4. 加入 Compaction（每 10 轮压缩一次），重复实验

5. 对比两组结果

预期发现：

- 无 Compaction：约 20-30 轮后开始"忘记"规则

- 有 Compaction：50 轮后仍能记住

实践 Demo 9：Lost in the Middle 实验

           return False, f"工具 {tool_name} 未被授权"

       for pattern in DENIED_PATTERNS:

           if pattern in str(args):

       def post_tool_use(self, tool_name, result):

实践 Demo 11：反馈循环 — 用数据驱动优化

目标：建立"记录 → 分析 → 优化"的闭环

步骤：

第一轮（基线测试）：

1. 准备 20 个不同的调研任务

2. 让 Agent 逐个完成

3. 人工评估每个任务的输出质量（1-5 分）

4. 记录成功率、平均分

第二轮（分析失败）：

1. 筛选得分低于 3 分的任务

2. 阅读对应的 trace 日志

3. 分类失败原因：

   □ 工具选择错误 → 优化工具 description

   □ 搜索结果不好 → 优化搜索查询的 prompt

   □ 上下文溢出 → 调整 compaction 策略

   □ 理解错误 → 优化 System Prompt

   □ 死循环 → 添加终止条件

第三轮（针对性优化）：

1. 根据分析结果修改 Harness 配置

2. 重新运行 20 个任务

3. 对比分数变化

预期效果：经过 2-3 轮迭代，成功率提升 20-40%

这就是 Harness Engineering 的核心循环。

七、学习原则总结

1. 从中间切入，不要自下而上

❌ 错误路线：Transformer 数学 → 预训练原理 → 微调 → ... → Agent

   （前 3 周全在看论文，没写一行代码，容易放弃）

✅ 正确路线：Tool Use → Agent → MCP → 遇到问题再回头补原理

   （第 1 周就有可运行的代码，持续有成就感）

2. Demo 驱动学习

每个概念都配一个具体的 Demo 项目

- 不是"看完教程觉得懂了" → 而是"亲手做出来验证理解"

- 不是"能跑就行" → 而是"故意测试边界情况"

3. 做减法比做加法重要

- 工具：5 个精心设计的 > 50 个随便写的

- Agent 步骤：让模型自己规划 > 你替它规定每一步

- 上下文：精选的 10K tokens > 塞满的 200K tokens

4. 错误是最好的老师

Agent 失败时不要急着修：

1. 先读 trace 日志，理解它为什么这样做

2. 分类失败原因（工具选错？上下文不够？死循环？）

3. 针对性修改一个地方

4. 重新测试，验证修改是否有效

5. 持续跟踪前沿

推荐关注：

- Anthropic Blog（每月读一篇）

- OpenAI Blog（每月读一篇）

- Simon Willison's Blog（AI 工程实践最佳作者）

- LangChain Blog（框架生态更新）

- MCP 官方规范 changelog

记住：好的 Harness 能让中等模型超越配备差 Harness 的顶级模型。 你的核心竞争力不在于用了多强的模型，而在于你构建的工具、上下文管理和反馈循环有多好。
