# 3.4 Multi-Agent

> 学习目标：理解 5 种协作模式，重点掌握 **Sub-Agent**，能判断**是否真的需要** Multi-Agent。

---

## 1. 底层逻辑：为什么 Multi-Agent 是反模式重灾区

Multi-Agent 听起来很酷，但**绝大多数场景下，一个设计良好的单 Agent 完胜一群 Agent 互相聊天**。原因有三：

- **上下文耦合**：Agent A 的输出是 Agent B 的输入，A 一抖动，B 全错；调试时你得同时盯两条上下文链。
- **通信开销**：Agent 间传话靠的是自然语言或 JSON，每跳一次都是一次"翻译损耗"，错误会累积放大。
- **调试黑盒**：单 Agent 出问题看一条 trace；多 Agent 出问题要拼链路、对时序、判断到底是谁的锅。颗粒度越细，越难定位。

**owner 意识**：先把单 Agent 做到极致（合理工具、清晰 prompt、合适上下文），证明它真的不够了，**再**上 Multi-Agent。这是顶层设计的纪律。

---

## 2. 判断框架（核心抓手）

```
我需要 Multi-Agent 吗？
├── 任务能拆成完全独立的子任务吗？
│   ├── 能 → 考虑并行 Multi-Agent
│   └── 不能 → 先用单 Agent
├── 单 Agent 上下文会溢出吗？
│   ├── 会 → 用 Sub-Agent 模式隔离上下文
│   └── 不会 → 不需要 Multi-Agent
└── 需要不同"专业角色"吗？
    ├── 需要（如写代码+审代码） → 考虑串行 Multi-Agent
    └── 不需要 → 不需要 Multi-Agent
```

> 三个问题里**任何一个回答"否"**，都先回去把单 Agent 打磨好。

## 3. 5 种协作模式

### 3.1 Sequential（串行流水线）

**形态**：A → B → C，每个 Agent 处理上一阶段产物。
**典型场景**：写代码 Agent → 审代码 Agent → 写测试 Agent。
**何时不要用**：上下游强耦合、信息会反复来回时——会变成低效的"传话游戏"。

### 3.2 Parallel（并行 fan-out / fan-in）

**形态**：一个分发器把任务切片 → N 个 Agent 同时跑 → 聚合器汇总。
**典型场景**：同时让 3 个 Agent 用不同策略写方案，再投票/合并。
**何时不要用**：子任务之间存在依赖（伪并行）、或聚合逻辑本身比子任务还复杂。

### 3.3 Sub-Agent（**重点**）

**形态**：父 Agent 在主流程中调用一个**独立上下文**的子 Agent，子 Agent 跑完只把"结论"返回给父。
**核心价值**：**上下文隔离**——子 Agent 看了 50 个文件、产生了 30k token 中间过程，父 Agent 只收到 200 字结论，主上下文不被污染。
**典型场景**：代码库探索、研究型搜索、长链路调试。
**何时不要用**：任务很简单、一两个工具调用就能解决——起子 Agent 反而是过度设计。

### 3.4 Router（意图分发）

**形态**：一个分类 Agent 判断意图 → 路由到对应的专业 Agent。
**典型场景**：客服系统（订单类→A、退款类→B、技术类→C）。
**何时不要用**：意图边界模糊、经常需要跨域协作时——Router 会反复来回甩单。

### 3.5 Hierarchical（管理者-执行者）

**形态**：管理 Agent 拆任务、派活、验收；下面一群执行 Agent 干活。
**典型场景**：大型多步骤项目（如 P9 拆需求给 P7 们）。
**何时不要用**：**没有明确终止条件**时——管理 Agent 容易陷入"再优化一轮"的死循环，烧钱烧时间。

## 4. Sub-Agent 深入（你需要重点掌握的）

### 4.1 它解决的核心问题

不是"让多个 AI 协作"，而是 **保护父 Agent 的上下文窗口**。

举例：你让 Agent 在 1000 个文件里找一个 bug。
- **不用 Sub-Agent**：父 Agent 自己读，读到第 200 个文件时上下文已被噪音填满，后面 800 个文件的判断质量直线下降。
- **用 Sub-Agent**：父 Agent 派一个子 Agent 去搜，子 Agent 读完 1000 个文件后只返回"bug 在 src/foo.py:42"——父 Agent 上下文干净，可以继续做后续决策。

### 4.2 与 Function Call 的边界

| 维度 | Function Call | Sub-Agent |
|---|---|---|
| 调用对象 | 确定性函数（API/SQL/计算） | 另一个 LLM 实例 |
| 输入输出 | 结构化（schema 严格） | 自然语言任务描述 |
| 决策能力 | 无（只执行） | 有（自主规划+多步） |
| 适用场景 | 单步、可枚举 | 多步、需要探索 |

**判断口诀**：能用 Function Call 解决就别起 Sub-Agent；起了 Sub-Agent，**任务描述要像写给一个新来的同事**，不能是一句话。

### 4.3 在 Claude Code 里的对应

- `Agent` tool 就是 Sub-Agent 模式的工程化实现。
- `subagent_type` 参数指定子 Agent 的人设（`Explore`、`general-purpose`、`Plan` 等）。
- 子 Agent 拥有**独立的上下文窗口**和**独立的工具集**。
- 父 Agent 只能看到子 Agent 的最终返回，看不到中间过程——这正是上下文隔离的体现。

**最小伪码**：

```python
# 父 Agent 主流程
result = call_subagent(
    type="Explore",
    prompt="在 src/ 下找出所有调用 deprecated_api() 的位置，"
           "返回文件路径+行号清单，不要返回代码内容。",
)
# result ~ 几百 token；子 Agent 内部可能消耗了几万 token
decide_next_step(result)
```

## 5. 反模式清单（踩过就是 3.25）

- ❌ **Agent 互相聊天的"会议室"设计**：3 个 Agent 围桌讨论，看着热闹，实际是同一个模型自己跟自己车轱辘话。直接用单 Agent + 多轮自我反思就够了。
- ❌ **把单 Agent 能做的拆成 3 个 Agent**：颗粒度过细 = 通信成本 > 收益。
- ❌ **Hierarchical 没有终止条件**：管理 Agent 永远觉得"还能再优化"，最后烧光预算。一定要有"完成判定"和"最大轮次"两道闸。
- ❌ **用 Multi-Agent 掩盖 Prompt 写得烂**：单 Agent 跑不好 → 起 Multi-Agent → 还是跑不好。根因是你 prompt 没写清楚，不是 Agent 数量不够。
- ❌ **没做可观测性就上 Multi-Agent**：链路看不见 = 出问题没法闭环。先把单 Agent 的 trace/log 做好。

## 6. 与 3.3 RAG 的衔接

RAG 系统天然适合做成 Sub-Agent：

- 父 Agent 收到用户问题，判断需要检索 → 调用"检索 Sub-Agent"。
- 检索 Sub-Agent 内部完成：query expansion → 向量检索 → reranker → 返回 top-k 片段+引用。
- 父 Agent 拿到精炼后的上下文，做最终生成。

这正是从单 Agent → Sub-Agent 的**自然过渡**：当你的 RAG 流程从"一次检索"长到"多轮检索 + multi-query + 自校验"时，把它打包成 Sub-Agent，主对话上下文就不会被检索中间产物污染。

> 在 [[RAG]] 里我们已经实现了一个完整的单 Agent RAG（检索 + 生成 + 失败模式分析）。下一步如果要做对话式 RAG / 多轮检索，就是 Sub-Agent 模式登场的时候。

---

## 7. 一句话总结

> **Multi-Agent 不是"用得多就高级"**：它的唯一合法用途是 ① 上下文隔离、② 真正独立的并行、③ 不可替代的角色分工。除此之外的 Multi-Agent，都是用复杂度换"看起来很厉害"——这是典型的反 owner 意识。

**先把单 Agent 做到 100 分，再谈 Multi-Agent。**
