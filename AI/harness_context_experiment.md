# Harness 对比实验：Context 管理策略

任务：长对话「早期信息保持」（3 组 needle，12 轮干扰，window=3）。
turn1 给关键设定 → 中间 12 轮无关闲聊撑大 context → 末轮提问，看哪种策略还记得。
模型：MiMo（mimo-v2.5-pro，reasoning 模型）。

## 修正后结果

| 策略 | 平均召回率 | 平均累计 prompt_tokens | 平均耗时 | 额外 LLM 调用 |
|---|---|---|---|---|
| **full** | 100% | 5,671（最贵） | 82s | 无 |
| **window** | 0% | 2,340（最省） | 78s | 无 |
| **rag** | 100% | 2,512 | 115s | 无（关键词检索） |
| **compaction** | 100% | 2,825 | 534s（最慢） | 每轮摘要 |

## 结论（怎么选）

- **full**：最准、最快，但 token 随对话线性膨胀，最贵——**短对话可以，长对话不可持续**。
- **window**：最省最快，但只留最近 K 轮，**早期信息直接失忆**（召回 0%）——适合「只关心近期上下文」的场景。
- **rag**：**综合赢家**。召回 100% + token 接近 window + 无额外 LLM 调用、耗时适中——适合「历史长、要按需翻旧账」的场景。
- **compaction**：召回 100%、token 省，但**慢得离谱（534s）**——因为每轮都要额外调 LLM 做摘要，且 reasoning 模型摘要本身慢。

## 两个重要注脚（别讲错）

1. **compaction 慢有我实现的锅**：本实验是 naive 版「每轮全量重摘旧历史」，越往后越慢；**生产级 compaction 是增量摘要（只摘新滑出的轮）**，会快很多。所以「compaction 慢」要打折看——它的真实代价是「需要额外 LLM 调用」，不是必然 534s。
2. **第一版 compaction 召回 0% 是 bug**：摘要的 `max_tokens=400` 被 reasoning 模型的推理吃光、content 返回空，导致设定全丢。**教训：reasoning 模型的任何 LLM 调用都要留足 max_tokens。** 修到 5000 后召回即 100%。

## 一句话选型

> 短对话用 full；只看近期用 window；**历史长又要记得住，首选 rag**；compaction 适合「能接受额外 LLM 成本、且做了增量摘要」的场景。
